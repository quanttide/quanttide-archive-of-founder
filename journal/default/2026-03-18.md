# 2026-03-18

今天找到了新的工具-范畴论。并且学习了哲学版本和数学版本。感觉非常有用，对于知识工程有了更透彻和本质的理解，感觉要有新的范式升级了。

打算试试 iflow，看看好不好用。
没备份成功昨天的日志。还好流程虽然经常断但是基本上都能恢复。这也不是坏事，从这些问题中能够发现哪些关键流程需要人或者平台接管。

meta 模块是很好的检验我的记忆模型的办法。journal 让它自己维护，report 按我需要的格式写了我看，tutorial 写了教我，等等。

归档是一个很有意思的 double check 流程。我发现这种多流转几个仓库的方法意外地起到了良好的抗丢失的效果，特别重要的还会出现在 usercase 里防丢。真的是很有意思的发现。

以前我不会这样工作，现在正在慢慢习惯这种低可靠性靠系统兜底的玩法。我在尝试接纳不完美，而是专注于创造工作中的核心价值。我觉得这或许可以让我自己能够越来越适应新方式。

慢慢造出来工具也可以让 AI 先用起来。维护元模块所需要的信息就是 AI 可以自己自动化的。我觉得这或许是一个很有意思的实验。

这些经验慢慢会增加我对于未来搭建系统的直觉，我觉得是有根本性的帮助的。

DeepSeek：“真正的智能不是永远正确，而是在不完美的世界中依然可以抓住核心。”

大幅简化了日志为 default 和 write 两类，一类随便写，一类要正式组织思路，这样可以区分清楚状态。

在考虑怎么分层认知，是从头开发还是直接复用。我比较担心复用依然解决不了我要白盒化的根本问题。我的白盒化就是能够在范畴论的框架下严谨地解释，我觉得这是我很重要的偏好。

git submodule 还是有指针设置的问题，总是踩这个坑。还是要验证是否提交成功，要么就是手动验证。我觉得或许这里也有什么范畴论的思想在里面。我发现范畴论特别适合适应动态变化，在动态变化里找相对稳定。

就是在混沌模糊的知识工作活动中找到特点活动分类。识别出来作为副活动。git submodule 或许就是一个可以尝试的方向。

有个主要的问题是，我们的设计方向是在快速变化的。比较担心写了白写。

感觉只在 default 里记录不去分流，总感觉工作没落实。但是从日志的表现来看，大部分工作也没有存的价值。我还是第一次通过日志的归档行为发现了这种变化。

我觉得我可以根据需要重启专题思考模式。从目前的需求来看，专题思考适合提出课题并且回答，很接近研究。

当整个系统越来越容错的时候，我们要把握的核心究竟是什么。我们该如何让 AI 落实我们的容错思想。

我觉得还是要从知识库的使用开始逐渐发现高价值需求，只靠自己是不够的。还是要不断的在生产活动中发现。

或许我可以从现成的流程出发呢，让团队给我开源知识库，我来根据这个整理出来新的代码，反过来再反馈。

这其实不是坏事，恰恰是我开始逐渐意识到了真正的视角和掌握它的方法。我的直觉告诉我，我的问题在于我的迭代需要非常快才能够快速进行海量的试错匹配业务流程，所以我选择了先升级工具。也是 AI 提醒我说这阻碍了真正的创造。所以是先创造再观察。因为只观察容易产生无意义感。只创造就。

尝试在团队建日志工作流。估计会比较困难。双核的好处是可以互相观察迭代。我可以不卡住协作流程先自己跑，也可以让团队慢慢跟我的流程。

这或许是智能迭代的 RL 模型，就是不断通过互相学习的良性竞争模式来进步。这是很好的竞争模型。

本来想做业务流程标准化和知识化的，现在感觉没有动力了。可能是还是觉得多多少少有点模糊。

想到了一个新概念，领域分层。之前做的是分类，不是技术上的，而是业务上的分级。这或许可以改变一些混乱。

比如领域层级，L0 就是被观察，L1 开始有自我指涉。

我现在的模式好像就是 meta 模式，就是纯粹从记忆模型出发标准化所有的工作。

个人的默认模式是日记，团队的默认模式是聊天。这两者之间的就是默认日志。最好是能够比较舒适地完成个人到团队的迁移。不过即使是一时半会做不出来也不影响。

我今天发现的另外一个意义是可以大规模地调用学生帮我们干活，这也是为量潮课堂的建设做铺垫。开源和显性化以后，经验可以更好的传递和铺垫。这样组织才能够越来越智能。

还有一个新的工作流，就是可以把公司观察我给正式化，哪怕是我自己切换成公司 CEO 身份观察创始人身份也是好的。

我在做的元标准是把自己的哲学翻译成数学来表达。大模型、小模型、规则引擎都可以互相验证。这样我就有完整的模型链了。

业务日志是管理后台视角，同样不是给客户看的，只有值得放档案里的才有可能需要给客户看。

一个关键改变。工作档案和工作报告换个位置，工作报告是未来事件记忆，工作档案是当下语义记忆。我原本想的知识就是工作档案该有的形态，所以这么改进就可以了。这样未来工程难度就会大大下降。
