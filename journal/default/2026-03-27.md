输入（混乱的现实）→ 分解（教学/工程的最小单元）→ 整合（可复用的体系）→ 输出（可传授的方法）

这是Kimi发现的我的基本思考方式。这就是CODE循环啊。而且真正的O是分解而不是组织。怪不得我之前总是找不到方向，原来是还需要改定义，把O变成分解。O的定义是分解整理。D是重新打包整合。先不改了，我直接强调分解和分门别类摆放就是组织就可以了。

创始人教程和公司教程最大的分工就是一个自上而下，一个自下而上回答具体问题。也不对。应该是当下和未来。

我突然有个大胆的想法，能不能通过小说讽刺母校的方式来表达意见。这样有创意有热度又不指代具体的人，只是人物难写一些。

这一段时间创作有点开始在状态了，逐渐找到了舒适区，就是坦诚地发泄就好了。理性的声音很多，可以一点点发。

和财务 HR 在采购 Code Plan 上的博弈，是对公司经营理念的争论。这个争论背后暴露的是当前公司非业务部门在经营理念上忽略隐形成本的问题。

我在尝试以工作教程的形式披露我的经营思路。首先要定性应当怎么经营，树立和明确各方都能经营的标准。

又出现了工作点分离的技术问题。先让助理帮我提交看看。我觉得我们的公司教程或许应该要带有特色，就是通过直接从 CEO 的角度讲来建立全局意识。手册是主要查阅的资料，所以流程用手册组织就可以。

我好像意识到了这种日记对于想法的消化作用。怪不得我的潜意识里总是提醒自己每天要写。虽然昨天晚上处理了这种因为工作摩擦产生的不适，也调节了心情，但是如果当时就写下来或许会更舒服一些。当然也不一定。

因为我们这种透明化的管理模式虽然带来了博弈的成本和困难，但是也给公开记录和讨论经营想法提供了基础。

这会带来什么还不确定。或许是更好和客户透明化呢。因为内部经营状况提炼出来就是要给客户讲的了。

这个事情在持续困扰我，虽然感觉在能力范围内，但是持续占用我的注意力。我觉得我先给记下来或许是个办法。如果有个思考云，那么感觉重点也许不在于要立刻分析，而是在于忠实地记录，帮助用户建立联想，而不是自己就给用户胡编乱造建议。
