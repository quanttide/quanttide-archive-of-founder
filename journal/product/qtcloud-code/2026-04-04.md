# 2026-04-04 编程云

首先要解决模块级的Python代码审计问题。比如变量、函数和模块等是否符合最佳实践。使用规则引擎判断，尽可能不使用大模型，等到需要的时候再用。

反向思考的核心原则，是“用最少的代码表达最多的业务问题”。如果用几百行代码只做了一件事情，那么业务意义比较有限。

命令设计大概是“qtcloud-code audit”，假设底层是OpenCode，主要是用来增强 OpenCode 的工程质量。难点是对于我们来说什么才算工程质量。前面的就算是我们的标准。

AI 回答了具体的想法，我问 AI，不考虑具体实现，我在现实中遇到了什么问题才会想到设计这些。AI 回答说，我遇到的问题是代码越来越臃肿，但是没人说的清什么才是好的业务代码，这样工程质量就会退化。而想要 AI 长时安全生产，就首先要定义清楚什么才是好的代码。

还是叫评审工具“review”命令。先实现我最头疼的圈复杂度问题。高层次的设计问题慢慢来。

Coding cli 实际上已经形成了一些事实表准。比如 review 命令普遍存在，但是通常只评审增量。那么还是 audit 存量比较合适。

我平时的一些固定动作可以封装。比如目前已经初步有了一个 Python 项目规范，要包括一些特定的文档。

在考虑把写作格式给转移到文档工程里定义。因为我发现特定格式需要结合 Markdown 解析器纠正语法。这样我们就做出来了一个“智能文档工程”。写作和文档的边界也更清晰。
