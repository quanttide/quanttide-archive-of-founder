# 2026-03-30

## HR云

目前没有像 Beancount 那样「官方 + 通用 + 纯文本 HR DSL」的成熟工具。
但已经有一些 “类 Beancount 思路”的 HR 相关项目，它们用纯文本 + 版本控制 + 可脚本化的方式来管理员工、工时、股权等。
1. 为什么 HR 领域没有“另一个 Beancount”
Beancount 的核心特征是：
纯文本、可编辑、可 diff
严格的双分录记账规则
丰富的查询、插件、导入生态
非常适合程序员自己理财、记账
HR 领域不太一样：
HR 信息更多是“表”而不是“账”：员工主数据、合同、薪酬档位、考勤、绩效、股权……
大部分公司需要的是 多人协作的 Web 界面 + 审批流 + 报表，而不是一个 CLI + 纯文本文件。
敏感度高，监管、隐私要求复杂，这也让“纯文本 + Git”这种方案更难直接落地。
所以现在更多是：
用“Beancount 思路”去改造部分 HR 数据，而不是做一个完整 HRIS 级的 Beancount。
2. 已经存在的“类 Beancount” HR 工具 / 实践
2.1 acc：一个纯文本 YAML 的“小 ERP”（含员工管理）
项目：72nd/acc，一个 CLI + YAML 的 plain-text ERP 系统
功能：
管理客户、员工、费用、发票
导入银行对账单（ISO 20022 camt）
能把业务数据自动生成 hledger 记账记录
特点：
数据存在 YAML 文件里，可以手工编辑，也可以用 CLI 交互式录入
适合小微企业整体业务数据管理，里面已经包含员工数据。
如果你想要的是“一个 Beancount 风格的 CLI + 文本，顺带把 HR 也管了”，acc 是目前最接近的一个。
2.2 Marathon：自托管、纯文本数据的员工管理工具
项目：connervieira/Marathon，开源、自托管的员工管理工具
功能：
员工信息管理
排班、打卡、工时记录
支付信息与工资计算
关键点：
所有数据都存储在纯文本文件中，便于管理员备份
自带 Web 界面，但底层数据是文本，可以用脚本、版本控制操作。
它不是像 Beancount 那种“一个文件搞定一切”，但底层数据就是纯文本，这是很典型的 Beancount 思路。
2.3 parti-time：纯文本工时记录（受 ledger / Beancount 启发）
文章：《It’s parti-time: Plain Text Time Tracking for Nerds》
思路：
作者认为传统工时工具太差，自己用受 ledger / Beancount / org-mode 启发的方式，设计了一个纯文本工时方案。
时间记录保存在 YAML 文件里，可以用 $ ti edit 调用编辑器修改。
特点：
纯文本 + CLI
可以导出 CSV / 报表
更适合个人或小团队记录工时，而不是完整 HR 系统。
如果你只是想“用 Beancount 的方式记工时”，这个方案可以直接参考。
2.4 Plaintext Equity：用 hledger / Beancount 管股权（cap table）
项目：Plaintext Equity
思路：
用一个 .ledger / .txt 文件记录所有股权变化，用 hledger / Beancount 做计算。
每笔股权变更都是一个记账条目，通过命令查询任意历史日期的股权比例。
文件本身就是一个完整的 cap table：
用文本 + git 做版本控制
所有变更可审计、可回溯。
这本质上是 用 Beancount/hledger 的工具链来做 HR 里的股权激励管理，非常符合你说的“类似 Beancount”的思路。
2.5 org-planning：文本化的组织结构与 headcount 规划
项目：8gg-git/knowledge-work-plugins 里的 org-planning 技能
目标：
输出文本化的组织结构图
做 headcount 规划、成本建模、招聘路线图
虽然它不是“一个通用软件”，但提供了一个思路：
用文本文件描述组织结构 & headcount 计划，
配合脚本/工具生成图表和报表。
2.6 Recutils：通用的“纯文本关系数据库”
工具：GNU Recutils，一个纯文本关系数据库系统
特点：
用 .rec 文件存放记录，每个字段 name: value，支持类型、key、约束等
提供命令行工具查询、过滤、join，类似 SQL 但针对文本。
用途：
完全可以用来设计一个“员工表”“合同表”“薪酬表”，构建自己的纯文本 HR 数据库。
配合脚本和版本控制，就能实现“类 Beancount”的 HR 管理。
3. 如果你想自己搭一套“HR 版 Beancount”，可以怎么做？
如果你主要看中的是 Beancount 的：
纯文本、可版本控制
可脚本化、可审计
用“账”的方式记录变化
可以考虑下面几种组合：
mindmap
  root((HR 版 Beancount))
    纯文本 ERP
      acc (CLI + YAML, 含员工/客户/发票)
      自建: Recutils 员工表
    工时与考勤
      parti-time (纯文本工时)
      Marathon (纯文本员工数据)
    股权与激励
      Plaintext Equity (hledger/Beancount cap table)
    组织与 headcount
      org-planning 文本化组织结构
3.1 一个可能的实践架构
员工主数据 + 合同
用 Recutils 设计几个记录类型：employee、contract、position 等。
放在 Git 仓库里，变更走 PR + 审批，等于自带审计日志。
薪酬/股权
薪酬结构（职级、薪资档位）可以用 YAML 或 Beancount 的 commodity 方式抽象。
股权/期权变更用 Plaintext Equity 的思路，用一个 .ledger 文件记录。
工时 & 考勤
小团队：直接用 parti-time 的纯文本方式记录工时。
稍微大一点：Marathon 这种自托管 + 纯文本底层的系统。
组织结构与 headcount 规划
参考 org-planning，用文本文件描述团队结构和 headcount 计划。
写脚本生成报表或图表。
4. 实际落地建议
如果你只是想“体验一下 HR 版 Beancount”：
从 Plaintext Equity 开始，用 hledger / Beancount 管理股权激励——这是最成熟、最直接的应用。
如果你是小团队、偏技术，可以尝试：
用 acc 管客户、员工、发票；
或者用 Recutils 自己搭一个“员工表 + 合同表”的纯文本 HR 数据库。
如果你有现成 HR 系统，只想“部分文本化”：
导出 CSV → 用脚本转为 .rec / YAML / Beancount 格式；
在 Git 中管理变更，写脚本做校验和报表。
总结一句话
目前没有一个像 Beancount 那样“官方 + 通用 + HR DSL”的成品，但已经有不少 用纯文本 + Beancount 思路管理员工、工时、股权的工具和实践，你可以按上面思路组合成一套“自己版本的 HR Beancount”。
