## 发现

充分利用大模型和小模型的分工。

在工程上的主要顾虑还是安全和记录，还是希望可以尽可能把大模型调用集中在一个本地入口，比如llm库一类的实现。如果实在需要自己实现也可以考虑。

### GLM

单一入口这个直觉是对的
为什么必须集中
LLM调用散落在代码各处，会同时失控三件事：
安全：哪个请求带了不该带的上下文，根本查不到
成本：哪个模块在偷偷烧token，无法审计
记录：没有日志就无法回溯”这个决策是大模型在什么上下文下做出的“
集中到一个入口，这三个问题变成一个问题的三个面。
这个入口本身该长什么样
你的”llm库“只需要做一件事：
def ask(prompt, context=None, model=”default“):
    log(prompt, context, model)   # 记录
    audit(prompt, context)        # 安全检查
    response = call(model, prompt, context)
    log(response)                 # 记录
    return response
所有业务代码只准调 ask()，不准直接调任何模型API。违反这个规则就是架构腐败。
和你”雪花编程法“的关系
这个 ask() 就是你说的雪核——所有大模型交互的唯一通道。后续无论换模型、加缓存、加限流、加审计，都只改这一个点，不用动业务代码。
又回到了同一个范式：先定锚点，再向外扩展。

### 小结

基本赞同这个思路，主要是怎么实现的问题。

智能体云的设计更多是调用，这里似乎需要一个infra层的解决方案，比如quanttide-llm二次适配，如果llm库不达预期。
