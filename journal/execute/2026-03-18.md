## 封装数据业务流程

如果我决定封装数据业务流程以方便观察和跟踪，我该怎么做？

我主要是担心分布式系统数据不同步，所以一直卡在了先做后台的问题上面。但是如果转到客户视角，问题就容易地多了。

我们可以选择先开客户端。在这个之前，可以先从业务出发，整体流程我是知道的，所以给手册化可能并不难。或者先写日志就可以。我觉得不是我的日志里没有含金量，而是没有处理流程容易断。不过这些都不是必须要做的，必须要做的还是要回到数据业务本身。

假设我们要给客户提供一个客户端进行全流程代替企业微信，我们可以先部署网页版给客户用。所以这个就是很好的观察对象。

如何从上面的梳理待办？
1. 是否要先写手册？不需要，手册是存在的，我也知道大概流程，或者可以自己凭空发明。
2. 是否需要先开完整项目？不需要，直接在外脑里试就好了，能交付就可以复用了。 

还可以顺手进行 GitHub org 的迁移，把业务和开源用不同的 GitHub org 隔离开，逐渐迁移过来。

是否考虑在thera的studio里尝试模拟设计完整的qtdata-studio的需求。还是先用 Jupyterbook 模拟效果。

写规则引擎才是首要的事情，之所以决定是因为发现了范畴论这个大杀器，给我处理复杂多变的业务规则提供了方法论，又可以和知识工程完美对接。
