### 1. 关于 commit 定位问题

这个仓库是 **元仓库 (Meta Repository)**，包含 10 个子模块：

| 子模块 | 定位 | 路径 |
|--------|------|------|
| **archive** | 过去-事件类：工作归档 | `./archive/` |
| **essay** | 未来-语义类：工作札记 | `./essay/` |
| **handbook** | 程序型-习惯法：工作手册 | `./handbook/` |
| **journal** | 现在-事件类：工作日志 | `./journal/` |
| **paper** | 程序型-权威法理：工作论文 | `./paper/` |
| **platform** | 程序型-宪法：技术平台 | `./platform/` |
| **profile** | 未来-事件类：工作档案 | `./profile/` |
| **report** | 现在-语义类：工作报告 | `./report/` |
| **specification** | 程序型-成文法：工程标准 | `./specification/` |
| **usercase** | 程序型-判例法：工作案例 | `./usercase/` |

每个子模块是独立仓库，看到的多条 commit 来自不同子模块。

**建议**：查看 `meta/metadata.md` 了解子模块状态和 Release 列表。

### 2. 关于交接 vs 己知 vs 不知

根据 `profile/think/default.md:102` 的框架：

- **对团队友好**：可交接、可培训、可验证 → 需要交接给团队
- **对AI友好**：有明确输入输出标准 → 内部使用
- **对你友好**：允许粗糙、允许碎片 → 个人知识

交接的核心是 **"管理职责交接制度"** (`profile/delib/org.md`)：
- 制度化目标：经验和责任能否物理转移到部门内部
- 关键抓手：可视化交接清单 + 试运行期

**简化理解**：
- 📤 **需交接**：handbook（工作手册）、specification（工程标准）、usercase（工作案例）— 可复用的制度性内容
- 📝 **个人记录**：journal（工作日志）、profile（工作档案）— 个人思考和过程
- 🔄 **中间态**：paper（工作论文）、essay（工作札记）— 整理后可交接
